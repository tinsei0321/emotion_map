### ADR-013 | 2026-06-17 | MCP 能力层纳入 vibe coding 体系 + 同类择优选智谱

3. **规则 11 改主备**：视觉识图主 = `zai-mcp-server`（智谱），备 = `vision-bridge`（火山引擎）。
4. **实测入档**：2026-06-17 全量冒烟测试 9 个服务，结果写入手册（7 通 / github 认证失败 / web-reader 重复）。

**后果**：

- ✅ 容错保留：智谱挂了有火山引擎 / 4_5v / skill 兜底

### ADR-012 | 2026-06-17 | 前端迁移：Streamlit → MapLibre GL JS (frontend/)

**状态**：✅ 已采纳

**背景**：
MVP 阶段前端基于 Streamlit（`apps/` :8501）+ Folium/pydeck。随产品演进暴露三个问题：
1. Streamlit 是数据应用框架，其 rerun 模型与全屏交互地图（拖拽/弹窗/实时筛选）天然冲突，状态管理反复踩坑（`@st.dialog` / `st.rerun` 规则复杂）。
2. Folium 是 Leaflet 的 Python 封装，定制深度有限，无法实现 geojson.io 级别的精细外壳（两层头栏 / 三栏 / 点击浮窗 / 底图 popover）。
3. 设计语言要从「毛玻璃/大圆角/暗底」切换到 geojson.io 简洁专业风格，Streamlit 的 chrome 难以彻底剥离。

**选项**：

| 选项 | 优点 | 缺点 |
|------|------|------|
| A. 继续 Streamlit + Folium | 零迁移成本、纯 Python | 交互天花板低、设计语言无法对齐 geojson.io |
| B. Streamlit + pydeck/组件库 | 仍纯 Python | pydeck 定制弱、Streamlit chrome 仍是束缚 |
| C. 独立前端 frontend/（MapLibre GL JS） | 交互自由、geojson.io 1:1、设计完全可控 | 引入 JS 栈、需 API 桥接数据 |

**决策**：
选择 C —— 新建 `frontend/`（纯 HTML/CSS/JS + MapLibre GL JS），geojson.io 1:1 外壳。
1. **MapLibre GL JS** 替代 Folium/pydeck：circle 层渲染情绪点 + 4 张天地图 WMTS 底图（CartoDB 因 CN 被墙移除）。
2. **配色单源**：`design/tokens.json` 新增 `geojson` 段，经 `generate_css.py` 生成 `frontend/css/tokens.css`（与原 Streamlit token 系统并存）。
3. **Streamlit 降级为迁移期遗留**：`apps/`（:8501）保留代码，仅维护不扩展；新功能一律进 `frontend/`；迁移收尾时下线。
4. **Phase 1** 已落地外壳（两层头栏 + 三栏 + 点击浮窗 + 底图切换），Playwright 全验证；Phase 2 接 `/api/v1/points` 真实数据。

**后果**：
- ✅ 交互天花板解除，geojson.io 设计语言 1:1 还原
- ✅ 配色单源未破（`tokens.json` 双套：原系统 + `geojson` 段）
- ✅ Streamlit 遗留代码保留，可回退、可对照
- ⚠️ 引入 JS 栈，前后端通过 API 桥接（Phase 2 待接 `/api/v1/points`）
- ⚠️ 两套前端并存期间，文档/配色/组件存在双轨，需持续去陈对齐（本次 9 份文档清理即为此）
- 🔜 迁移收尾：Streamlit 下线、folium/pydeck 依赖移除、配色单源合并

### ADR-011 | 2026-06-13 | 引入决策追踪系统 (Decision Tracking System)

**状态**：✅ 已采纳

**背景**：
传统调试方式靠"读代码+猜位置"——在报错后通读源码搜索关键词，是 O(n) 复杂度。
随着项目规模增长（当前 13 个核心 .py 文件、11 个 Agent），bug 定位成本线性上升。
需要一种架构级的方案，让 bug 定位从"大海捞针"变为"按图索骥"。

**选项**：

| 选项 | 优点 | 缺点 |
|------|------|------|
| A. 传统调试（print+搜索） | 零成本上手 | O(n) 复杂度，随代码量线性恶化 |
| B. 第三方 APM（Sentry/DataDog） | 功能完善 | 需外部服务、付费、过度设计 |
| C. 自研决策追踪系统 | 零依赖、贴合项目、O(1) 定位 | 需自行维护、需全员埋点 |

**决策**：
选择 C（自研决策追踪系统 `core/tracker.py`），因为：
1. 项目规模适中，自研成本可控（~300 行）
2. 每个功能/行为/代码块分配唯一决策 ID（MOD_XXX.F_NNN / MOD_XXX.D_NNN）
3. 运行时自动记录 enter/exit/error 日志，形成完整调用链
4. 配合 Debugger Agent 的新工作流：看 [TRACE] 日志 → 定位 ID → 跳转代码

**核心设计**：
- **非侵入式**：装饰器 (`@track`) + 上下文管理器 (`TrackContext`) + 快捷函数 (`trace_*`)
- **层级 ID**：模块→函数→决策点 三级编号
- **零成本关闭**：`TRACKING_ENABLED=False` 全局关闭
- **结构化日志**：统一 `[TRACE]` 前缀，便于 grep/解析
- **注册表**：所有 ID 在 `core/tracker.py` 中登记，代码即文档

**后果**：
- ✅ bug 定位从 O(n) → O(1)，指数级提升 debug 效率
- ✅ 追踪日志天然形成数据管道审计链
- ✅ 新成员通过注册表快速了解代码结构
- ⚠️ 需要全员遵守埋点规范（铁律 9/10）
- ⚠️ 现有代码需逐步补埋追踪点（非一次性，渐进式）
- 🔜 未来可扩展：日志写入文件 + 可视化调用链 + 性能剖析

### ADR-001 | 2026-05-30 | 选择 SnowNLP 作为 MVP 情绪分析引擎

**状态**：✅ 已采纳

**背景**：
需要在 MVP 阶段快速实现文本情绪分析。备选方案包括 SnowNLP、Senta（百度）、BERT-wwm-ext。

**选项**：

| 选项 | 优点 | 缺点 |
|------|------|------|
| SnowNLP | 轻量(~10MB)、离线、pip 安装、中文友好 | 精度一般、仅正面/负面分数、无情绪分类 |
| Senta (百度) | 精度高、支持细粒度 | 需 PaddlePaddle 框架、模型较大、部署复杂 |
| BERT-wwm-ext | 精度最高 | 需 GPU、模型巨大(~400MB)、零基础难上手 |

**决策**：
选择 SnowNLP。MVP 阶段追求"快速可用"而非"精度最优"。通过 `AnalyzerBase` 抽象接口预留了未来升级路径。

**后果**：
- ✅ 分钟级完成环境搭建和首批数据分析
- ✅ SnowNLP 的不准确性可转化为语料库本地化研究的切入点
- ⚠️ 精度限制可能导致部分分析结果偏差，需要在展示层加以说明
- 🔜 未来通过 `LLMAnalyzer` 接口平滑升级到溯佰科 LLM

### ADR-007 | 2026-06-12 | 系统架构层级优化：新增数据采集层 + 空间分析引擎重定义

**状态**：✅ 已采纳

**背景**：
项目架构最初为五层（应用层 / UI组件层 / 地图引擎层 / 分析引擎层 / 基础设施层 / 数据层）。随着对项目理解的深入，发现以下不足：
1. 缺少数据采集层——数据爬取是独立职责，不应混入其他层
2. "地图引擎层"概念过窄——未来需要缓冲区分析、空间聚合等 QGIS 级别功能
3. "分析引擎层"命名不够精确——该层实际承担 raw→L1(治理)→L2/L3/L4(分析) 的完整数据加工职责
4. 溯佰科定位错误——在代码文档中被描述为"LLM大模型"，实为城市规划时空大模型平台

**决策**：
1. **新增"数据采集层"（SCRAPER/）**，置于数据层之上，负责多源数据爬取（大众点评/美团/小红书/微博/12345），基于 Scrapy 框架，备用方案为购买数据
2. **"地图引擎层"→"空间分析引擎层"**，扩展职责为：底图渲染 + 空间可视化 + 空间分析（热点分析/缓冲区分析/行政单元聚合），MVP 阶段空间分析工具选型 geopandas + shapely 自研
3. **"分析引擎层"→"数据分析引擎层"**，明确其 L1(数据治理)→L2(SnowNLP)→L3(LLM)→L4(多维归因) 四级数据加工管道
4. **纠正溯佰科定位**：从"LLM大模型"改为"城市规划时空大模型平台（数据底座+GIS工具+NL工作台）"，情绪地图未来以 Agent 方式嵌入
5. **数据层**增加 L0~L4 五级数据分级概念，导出命名规范对应

**后果**：
- ✅ 架构层级从 6 层扩展为 7 层，职责更清晰
- ✅ 空间分析引擎层有了明确的功能路线图和工具选型建议
- ✅ 溯佰科定位纠正，避免未来开发方向偏差
- ✅ 数据分级(L0~L4)与导出命名规范一致，溯源清晰
- ⚠️ 数据采集层技术方案尚未最终确定（Scrapy vs 购买），需近期明确
- ⚠️ 空间分析核心功能（缓冲区分析/行政单元聚合）需编码实现

### ADR-008 | 2026-06-12 | 选择 Scrapy 作为数据采集框架

**状态**：✅ 已采纳

**背景**：
需要从大众点评、美团、小红书、微博、12345 等多个中文平台持续采集城市情绪相关文本数据。面临三个选项：自研脚本、Scrapy 框架、购买数据。

**选项**：

| 选项 | 优点 | 缺点 |
|------|------|------|
| 自研脚本 (requests+bs4) | 上手快、几十行能跑 | 每个网站单独开发、反爬/去重/断点续抓需手写 |
| Scrapy 框架 | 内置去重/限速/Pipeline、多源管理统一、社区成熟 | 学习曲线（需理解 Spider/Engine/Pipeline 概念） |
| 购买数据 | 零技术负担、合规 | 成本高、依赖供应商、数据定制性差 |

**决策**：
选择 Scrapy 框架。理由：
1. 5 个数据源需要持续采集，不是一次性任务
2. 内置的去重、断点续抓、自动限速等功能省去大量手写代码
3. Pipeline 架构与项目现有的 Pipe-and-Filter 设计一脉相承
4. 备用方案：若爬取严重受阻，购买数据作为兜底

**后果**：
- ✅ 项目搭建完成：9 个文件、标准 Scrapy 结构、首个 Spider（小红书）
- ✅ 测试验证：小红书搜索页 HTTP 200，SSR 数据可无登录提取
- ⚠️ 各平台反爬策略不同，需逐个适配
- ⚠️ 购买数据仍为备用方案，不排除未来切换

**背景**：
v1.0 的 11 Agent 手动编排模式存在三个问题：
1. 用户需要手动逐一切换 Agent（`@pm` → `@developer` → `@reviewer`），纯手动开销大
2. Agent 数量偏多，存在职责重叠（Debugger/Reviewer、Design Reviewer/Designer）
3. Agent 定义完整但未真正被自动调用——是"给人看的 SOP"而非"自动执行的编排"

**决策**：
1. **精简阵容**：11→8 Agent
   - Debugger 并入 Developer（Developer 同时具备诊断修复能力）
   - Design Reviewer 并入 Designer（自审清单把关，交付前自查）
   - PM 角色由 Claude Code 主线程承担（不再独立 Agent）
2. **自动编排**：用户一句话给任务，Claude 内部自动 spawn Agent 走 SOP
3. **保留独立**：Developer / Reviewer / Tester / Designer / Data / GIS Dev / Docs / Ops

**后果**：
- ✅ 用户不再需要手动 `@agent` 切换，任务启动成本降为零
- ✅ Agent 数量减少 27%，角色边界更清晰
- ✅ Developer 诊断能力增强（基于追踪系统的 O(1) 定位）
- ✅ Designer 自审清单确保交付质量不降低
- ⚠️ Developer 职责更重（开发+调试），需要更全面的能力
- ⚠️ Designer 自审不能完全替代独立审查，重大 UI 改动仍需额外把关
