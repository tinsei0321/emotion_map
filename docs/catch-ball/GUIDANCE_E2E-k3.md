## 〇、总纲：端到端 = 四条链的闭环

v1.0 已把「装载链」（import→range→layers→analyze）和「交互环」（点击要素→分析→归因→地图定位）设计收敛。端到端体验的完整定义应再补两条链，形成四链闭环：

| 链 | 内容 | v1.0 覆盖 | 本文补强 |
|---|---|---|---|
| ① 装载链 | 数据进场→范围→图层 | ✅ 真值表 + 光环/banner | 数据就绪度升维（§2.2） |
| ② 交互环 | 点击要素→分析→归因→回视野 | ✅ S3 空间优先 + S4 定位 CTA | 三端同步 2.0（§3-F4）、张力雷达（§3-F3） |
| ③ 诊断链 | 问题→意图→工具→四态出口→呈现范式 | ⚠️ harness 内已有但呈现端未范式化 | 出口呈现范式（§2.4）、尺度护栏前置（§2.2/§3-F7） |
| ④ 产出链 | 结论→导出→分享→再进入 | ⚠️ 仅 export CTA | 气象播报（§3-F2）、演示模式（§3-F1）、反馈闭环（§2.5） |

**北极星不变**：演示逻辑链（张力图面→引导点击→交互分析→宏观诊断）+ 三端同步铁律（视野↔数据↔结论）。本文所有建议逐条标注所服务的链与环节。

## 一、架构设计

### 1.1 三层编排架构（信号 → 编排 → 执行）

v1.0 的 cpd-state（信号）/ cpd-guide（编排）/ 既有入口（执行）已是正确分层。建议把这层关系**制度化**，防后续演进坍塌回"大杂烩 panel.js"：

```
┌─ 信号层 cpd-state.js ────────────────────────────┐
│  谓词注册表：hasImport / hasRange / hasVisible-  │
│  EmotionLayer / hasAnalysis / isBusy / streaming │
│  每个谓词 = 纯函数 + 判据来源注释 + 单测          │
└──────────────┬───────────────────────────────────┘
               │ 特征向量（只读快照）
┌──────────────▼───────────────────────────────────┐
│ 编排层 cpd-guide.js（v1.0 已定稿：确定性真值表）  │
│  演进位：真值表外置为 JSON 策略表（§1.4）         │
└──────────────┬───────────────────────────────────┘
               │ guidance 对象（kind/text/target）
┌──────────────▼───────────────────────────────────┐
│ 执行层（全部既有，不重造）                         │
│  panel.js 渲染 / sidebar.js cpd:guide-cta /      │
│  map focus / openImport / *_dialog / _exportReport│
└───────────────────────────────────────────────────┘
```

**建议 A1（谓词注册表 + 谓词级测试）**：CB-CPD-01 的 `.aiq-conclusion` 死信号、CB-CPD-03 的 `hasVisibleEmotionLayer` 谓词盲区，根因都是"谓词没有独立测试"。落法：每个谓词导出为命名纯函数，Playwright 行为用例直接断言谓词真值（复用 `tests/browser/lib/emc_helpers.py`），把"死信号"从评审发现变成测试发现。**服务链：①②的可靠性地基。成本：G1 顺带做，几乎零额外。**

**建议 A2（事件命名空间规范）**：现有 `cpd:turn-ended` / `cpd:focus-tab` / `cpd:guide-cta` / `cpd:reset` + `layers:changed` / `layer:selected` 两类并存。建议明文规范：CPD 域事件一律 `cpd:` 前缀、载荷 `{type, payload, meta{ts, source}}`；地图/图层域保持原样。文档落 `docs/design-system.md` §4（v1.0 §九已列该文件 G4 更新）。**成本：一段文档。防后续事件野蛮生长。**

### 1.2 上下文织体（Context Fabric）——LLM 与引导引擎共用同一事实源

现状：`buildContext`（tools.js:455-458）已把 curState 注入 LLM 语境（"仅供参考，不改变工具选型"），引导引擎则从谓词自行推导。**两个消费者、两套取数路径**，长期必漂移。

建议：把"会话事实快照"抽成单一生产者——

```js
// cpd-state.js 扩展（G3 演进位）
export function getSessionSnapshot() {
  return {
    range: /* isRange 层摘要（名称/几何类型） */,
    layers: /* 可见情绪层清单 + 数据覆盖摘要 */,
    timepoint: /* time-bar 当前时点 */,
    lastExit, lastRegion: /* _followUps:455 同源 */,
    readiness: /* §2.2 数据就绪度向量 */,
  };
}
```

- **引导引擎**消费它算 guidance（替代散装谓词调用）；
- **buildContext** 消费它生成 LLM 语境（替代 tools.js:455-458 的散装拼接）。

承重兼容性：✅ 快照对 LLM 仍是"仅供参考"注入，不参与 diagnose 路由裁定（v1.0 §七措辞继承）；引擎仍纯客户端。收益：引导文案与 LLM 回答**永远引用同一份事实**（"两个模块说不一样的话"这类脱节从结构上消失）。**服务链：①③。时机：G3（G1 先用散装谓词 ship，G3 收编）。**

### 1.3 引导策略表外置（远期，G4 后评估）

真值表目前硬编码在 `deriveGuidance()`。建议 G4 后评估外置为 `frontend/js/ai_qa/cpd-policy.json`：行 = 特征条件 + guidance kind + 文案 key。收益：① 调引导策略不改代码、可进版本管理 diff；② 文案集中一处，叙事化打磨（v1.0 §4.2"文案 G1/G2 打磨"）有固定落点；③ 为 §2.5 反馈闭环的"引导策略调优"提供数据驱动的抓手（改 JSON 比重构函数安全）。**不提前做：G1-G2 真值表仍在震荡期，过早外置是浪费。**

### 1.4 架构上的三个"不要"

1. **不要**让引导引擎长成第二个 harness——编排层永远确定性、永远不进 LLM（v1.0 决策 1 继承）。需要"更聪明的引导"时，改策略表数据，不加 LLM 调用。
2. **不要**为端到端新建"页面"或"模式切换"——端到端 = 同一界面上引导密度的变化，不是另一套 UI。演示模式（§3-F1）也是复用同一引导引擎的脚本化，不是新页面。
3. **不要**碰 harness/stages/tools/diagnose/四态出口（v1.0 承重全文继承）。本文所有机制建议的落点都在：客户端引导层、review 层、buildContext 注入端、episodes 消费端。

## 二、核心机制

### 2.1 用户意图理解：三层结构 + 意图链

现状两层：`_quickIntent`（本地轻量预判）→ `diagnoseStep`（LLM 诊断卡，永不动）。建议明确为三层并补齐会话级维度：

| 层 | 机制 | 现状 | 建议 |
|---|---|---|---|
| L0 即时预判 | `_quickIntent` 本地规则 | 已有 | 从 episodes 挖掘高频问法扩规则库（§2.3），覆盖越多越省 token |
| L1 诊断卡 | diagnoseStep | **永不动** | — |
| L2 意图链 | 跨轮意图追踪 | 雏形已有（`ctx.resume` 续作，harness.js:400-405） | 显性化：维护"本会话意图链"（连续 3 轮同域 = 深挖态；突然换域 = 新任务态），供两处消费——buildContext 注入（"用户正在深挖 X 区"）+ 引导引擎（深挖态不推 range/import 类装载引导，改推深读/对比） |

意图链的客户端部分是纯信号（不动 diagnose），落 cpd-state 快照（§1.2）。**服务链：③。时机：G3。**

### 2.2 数据处理：从 hasImport 到"数据就绪度"（readiness vector）

v1.0 的 `hasImport` 是布尔——"有没有"。端到端体验需要"够不够"：

```js
readiness = {
  coverage: 'none' | 'partial' | 'good',   // 范围内数据点密度 vs 阈值
  timepoints: 1 | 2 | 3,                    // 可用时点数（T1-T3）
  scale: 'district' | 'subdistrict',        // 数据可支撑的最细尺度
}
```

三个消费点：
1. **引导文案升维**：「生成第一张情绪地图」→「已导入 12,847 条，覆盖西陵区——框选范围看张力」（确定性数据，客户端可算，不违反"动态变量无源"教训——这是图层元数据，不是 LLM 输出）；
2. **尺度护栏前置（F7 的机制底座）**：`readiness.scale` 告诉前端"数据只支撑到区级"，用户提问含微观词时在**输入阶段**轻提示（详见 §3-F7）——尺度诚实从"事后 review 纠"（配套 A）前置为"事前 nudge"，两者互补不冲突；
3. **GAP 出口的内容源**：gap 时 EMC 说"缺什么"——readiness 让这句话从 LLM 推测变成客户端事实（注入 buildContext）。

**服务链：①③。时机：G2-G3（readiness 计算依赖图层元数据，G1 先布尔 ship）。**

### 2.3 语义识别与映射：从 LLM 独扛到"LLM + 本地模式库"双轨

现状：自然语言→GIS 操作全部由 LLM 路由（diagnose→SKILL_DEFS→TOOLS）。端到端要更快、更省、更可测：

- **挖掘**：`ai_qa/consolidate.py` 已在做 L3 episodes→L2 wisdom 的周期挖掘（api/main.py:81-82 三层闭环已存在）。新增一个挖掘维度：**高频"问法→工具序列"模式**（如"哪里情绪最差"→filter+rank、"X 周边怎么样"→buffer+zonal）。
- **沉淀**：命中 ≥N 次的模式固化为 `_quickIntent` 本地规则（带置信度标注），本地直路由，跳过 diagnose——**省 token、降时延、且本地规则可进单测**（vs LLM 路由只能运行时抽验）。
- **地名词典**：宜昌片区别名/俗称（"CBD"="夷陵广场商圈"类）客户端词典化，L0 层先做地名归一再进路由。数据源同样从 episodes 挖掘。
- **承重**：✅ diagnose 不动——本地规则只处理"极高置信捷径"，未命中仍走完整 diagnose；这与现有 `_quickIntent` 定位一致，只是扩充其规则库。

**服务链：③。时机：P2 后持续运营（episodes 积累量决定质量，先有数据再挖掘）。**

### 2.4 回答范式：四态出口的"呈现范式"标准化

四态出口（+ general）不动，但每种出口的**呈现**应标准化——用户形成稳定预期，演示时节奏一致：

| 出口 | 呈现范式 | 关键组件 |
|---|---|---|
| `result` | **诊断卡**：顶部极性色带 + 结论段 + 内联情绪色块（emotion-swatch）+ 建议动作组 + 地图定位 CTA | v1.0 §4.2 row5 + design-system 三级权重一级 |
| `gap` | **数据清单卡**：缺什么（readiness 事实，§2.2）+ 上传入口 CTA + "用现有数据先答"次 CTA | _followUps:457-462 已有胶囊，升级为卡 |
| `partial` | **半成卡**：已完成部分照常渲染 + 待补清单折叠 + 补完 CTA | _followUps:471-476 升级 |
| `ask` | **选项胶囊**（现状已好） | onAskUser 渲染，不动 |
| `drift` | **拦截卡**：说明 + 换问法/缩小范围 CTA | _followUps:465-469 升级 |
| general | **轻回答**：无卡无 CTA，纯文本 + 可选追问胶囊 | 现状已好 |

统一规则：卡片全部走三级权重体系（结论卡浮出/过程卡折叠/审查胶囊），颜色全 theme var。**服务链：③④。时机：G2（banner 体系建立时顺带统一）。**

### 2.5 反馈闭环：三件套（已有底座的完整化）

L1=MANIFESTO / L2=wisdom.py / L3=episodes.jsonl 三层闭环已存在（api/main.py:81-82；/aiqa/episode fire-and-forget，panel.js:1188；consolidate.py 周期挖掘）。缺的是消费端三件套：

1. **回放评估（Replay Eval）**：定期取 episodes.jsonl 历史问法重跑 harness，对比新旧回答的出口分布/审查通过率——**防 prompt/review 改动引入回归**（与 tests/eval_template_flash.py 的 diagnose 路由 eval 互补：eval 测路由，回放测端到端质量）。落 tests/browser 或独立脚本，人工触发。
2. **引导漏斗遥测**：客户端计数 `guidance.shown / engaged / completed / dismissed`（按 kind 分桶），localStorage 聚合、导出报告时附带。**引导策略调优从拍脑袋变数据驱动**——哪个引导文案无人点、哪个状态用户卡住，一目了然。承重：✅ 纯客户端计数，不进 LLM、不写后端。
3. **结论卡显式反馈**：诊断卡角落 👍/👎（极简），回写 /aiqa/episode（载荷加 `feedback` 字段，episode.py 加字段——后端小改，非承重清单内）。👎 样本进 consolidate 优先挖掘队列。

**服务链：④+全链调优。时机：遥测 G2 顺带（成本低）；回放评估 P2 后；显式反馈 G4。**

## 三、关键 Feature（创新 · 服务演示链）

> 每个 Feature 标注：【服务链环节】【依赖阶段】【承重兼容性】。排序 = 建议优先级。

### F1 演示模式（Demo Script Mode）——"产品自己讲自己的故事"⭐ 旗舰

一键进入脚本化演示：按演示逻辑链自动编排——载入示例数据（sim 底座已有）→ 张力图面亮起 → 地图自动聚焦最深橙区域（呼吸高亮）→ EMC 自动"问"出预设问题并真实跑 harness 回答 → 结论卡浮现 → 时间轴播放演变 → 导出报告收尾。**全程真实数据真实计算，不是录屏不是幻灯片。**

- 实现：引导引擎的"脚本化"变体——预排 guidance 序列 + 自动 CTA 触发，复用 cpd-guide/真值表/CTA 调度全套既有机制，**不是新页面新引擎**（§1.4-2 的约束）。
- 业界对标：BI 工具的 presentation mode 全是静态快照；**live 数据驱动叙事 + AI 现场分析**在 GIS/BI 产品中未见。演示场景（规划局汇报）正是本项目核心场景。
- 【交互环+产出链全程】【依赖 G3（CTA 调度成熟后）】【✅ 复用引导引擎，承重零触】

### F2 情绪气象播报（Emotion Weather Report）——宏观诊断的公众语言

S4 结论一键生成"城市情绪气象"播报卡：「今日宜昌情绪天气：**局部有怨**——西陵区深橙聚集（老旧小区改造话题），点军区晴（滨江公园正面情绪稳定），整体情绪指数 62，较上周 ↓5」。卡片可导出图片分享。

- 价值：把"宏观主观诊断透镜"的产品定位翻译成**任何人 3 秒能懂**的语言；演示时的"哇"时刻；社交传播入口。
- 实现：结论 trace 已有区域/极性/样本量结构化字段（diagnose.card + review），模板化生成（不走额外 LLM 调用，或可选 LLM 润色一档）。
- 【产出链·结论→分享】【G4】【✅ 读 trace 结构字段，承重零触】

### F3 张力雷达（Tension Radar）——常驻的"张力可见性"

地图角落常驻迷你面板：当前视野内情绪分布极性条（深绿/浅绿/灰/浅橙/深橙 五段占比条形，随视野移动实时重算）+ 最深橙区域名。**张力图面不再是"要用户自己看出来"，而是常驻可量化。**

- 价值：演示逻辑链第一环（张力图面）的放大器；演讲者不用口头指"你看这里很红"，雷达自己说话。
- 实现：纯客户端——监听 map moveend → 取视野内点要素的情绪字段 → 五段计数。色值全走 theme var 情绪端点。
- 【交互环·张力图面】【G2（与 banner 体系同期，布局一起排）】【✅ 只读图层数据】

### F4 三端同步 2.0——"散文句子级"联动

既有铁律（视野↔数据↔结论橙色 #ff9000 同步）的完全体：
- 正向：hover 结论卡中**区域名/数据句** → 地图对应要素呼吸高亮 + Overview 对应行高亮；
- 反向：hover 地图要素 → 结论卡中**提到该区域的句子**底色高亮（不是只高亮卡片标题，是定位到句子）。

- 实现：结论卡渲染时把 `[ref:区域]` 标注（review.py data_driven 已要求 [ref:区域名]，review.py:42）包成 `<span data-region>`；地图 hover 要素时按 region 名反查 span。#ff9000 一并 token 化（CB-CPD-03 L2 遗留）。
- 业界对标：BI 联动停留在图表↔图表；**文本句子↔地图要素**的语义级联动是差异化。
- 【交互环核心】【G3】【✅ 渲染层增强，承重零触】

### F5 时光对比叙事（Timeline Story）——时间轴变叙事播放器

T1→T3 切换时自动生成一句演变叙事（「负面情绪从西陵老城向伍家岗扩散」）浮在时间轴旁；加播放键：自动轮播时点 + 逐句解说 + 地图同步变化。

- 实现：时点对比是确定性计算（两时点同区域极性差），客户端可算；叙事句模板化。time-bar.js/timeline.js 增强，非重写。
- 【交互环+产出链】【G3（timeline 轻编排同期）】【✅ 纯客户端】

### F6 尺度护栏（Scale Guardrail）——尺度诚实的事前形态

输入框实时检测微观尺度词（"某条街/某栋/精确到米"）→ 输入框下方轻提示：「街道级精度有限，建议问片区趋势——我可以告诉你 X 街道所属片区的宏观情绪」。用户可忽略照常发送（软护栏，不阻断）。

- 与配套 A 关系：配套 A = 事后 review 纠（回答层）；F7 = 事前 nudge（输入层）。**两道防线互补**，且 F7 不触发任何 LLM 调用（客户端正则 + readiness.scale，§2.2）。
- 【诊断链·入口】【G3（依赖 readiness）】【✅ 纯客户端，diagnose 零触】

### F7 引导漏斗遥测面板（内部向）

开发者模式下可见 guidance 漏斗面板（§2.5-2 的可视化）。自己用的调优工具，顺手即是演示时"我们怎么迭代引导"的故事素材。【全链调优】【G2 顺带】【✅】

### F8 What-if 沙盘（远期·抛光期评估）——"如果在这里建公园"

拖拽虚拟设施（公园/商业体）到深橙区域 → 基于 sim 模拟底座重算周边情绪 →「模拟：此地建公园后，周边 500m 情绪或改善一档」。**规划决策的"所想即所见"**，演示震撼力天花板。
