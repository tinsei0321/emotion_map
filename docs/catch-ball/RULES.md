# Catch-Ball 评估规则与工作指南

> **版本**：1.0 | **生效日期**：2026-07-19
> **适用对象**：第三方评估模型（ZCode + DeepSeek V4 Pro）
> **评估项目**：emotion_map — 城市情绪地图平台

---

## 一、评估原则

### 1.1 第三方中立
- 评估方不为项目方的任何决策背书或辩护
- 不预设结论，一切判断基于扫描证据
- 对项目的优势和问题同等重视，不偏袒

### 1.2 证据驱动
- 每个判断必须附带文件路径引用（尽可能到行号）
- 评分必须有可追溯的具体发现支撑
- 不依赖项目方的自我评价——独立核实

### 1.3 建设性批评
- 发现问题必须附带可操作的改进建议
- 建议按严重程度和影响面分级
- 讨论点聚焦架构级话题，避免陷入细节争论

### 1.4 双模型闭环
```
SCAN（DeepSeek 评价）→ CB Journal（Claude Code 反评价 + 行动）
    → 下一轮 SCAN（对比验证改进效果）→ 持续迭代
```
- 每轮 SCAN 首章必须回顾上一轮建议的执行情况
- 对比验证是闭环的核心价值——衡量建议是否落地产生了实际改善

## 二、评估方法

### 2.1 六轴评分体系

| 轴 | 权重 | 评估内容 |
|----|------|---------|
| **架构设计** | 20% | 分层清晰度、模块边界、接口契约、扩展性、技术选型合理性 |
| **代码质量** | 25% | 可读性、DRY 原则、错误处理、追踪覆盖率、安全实践 |
| **测试覆盖** | 15% | 单元测试数量与质量、E2E 覆盖、边界条件测试、CI 集成 |
| **Harness 工程** | 20% | Agent 体系完善度、Hook/Command 有效性、SOP 风险分级、自动化门禁 |
| **文档完整度** | 10% | PRD/Spec/ADR/开发日志的同步度、准确性、可发现性 |
| **调用效率** | 10% | Agent spawn 次数合理性、Token 消耗、上下文加载效率 |

> 综合分 = 各轴得分 × 权重 之和（四舍五入到一位小数）

### 2.2 三级扫描深度

| 级别 | 阅读范围 | 适用对象 |
|------|---------|---------|
| **L1（全量阅读）** | 逐行阅读全部代码/配置 | 核心配置（AGENTS.md, CLAUDE.md）、所有 Agent 定义、所有 Hook 脚本、tracker.py、config.py、ai_qa/llm.py、frontend/serve.py |
| **L2（关键函数+结构）** | 阅读函数签名、关键逻辑、文件结构 | 所有 Core 模块、所有 API 路由、AI QA 核心文件、Frontend 主要 JS 模块 |
| **L3（概览+头部）** | 阅读文件头部 docstring、类/函数清单、文件大小 | SCRIPT 分析脚本、测试文件、文档文件、Scrapy 爬虫 |

### 2.3 八维评估子维度

1. **Vibe Coding 策略**：AI 辅助开发体系（Agent 协作、SOP、铁律）的合理性与有效性
2. **Harness 框架**：Hook/Command/自动化门禁的设计质量与覆盖度
3. **Agent 体系**：角色定义质量、协作协议、可发现性
4. **Skills 体系**：技能库的精选质量、落地率、触发规则
5. **架构设计**：分层合理性、模块边界、技术债
6. **代码质量**：逐模块评级（A/B/C）+ 跨模块共性问题
7. **推进情况**：各子系统完成度估算 + 最新进展
8. **调用消耗**：Agent spawn 频次、Token 消耗分析

## 三、Catch-Ball 环节定义

### 3.1 环节流程图

```
CB-N 轮开始
    │
    ├── ① SCAN（第三方）
    │   ├── 回顾 CB-(N-1) 建议执行情况
    │   ├── 全面扫描项目当前状态
    │   ├── 六轴评分 + 趋势对比
    │   ├── 提出分级优化建议
    │   └── 输出 SCAN_DeepSeek_{NN}.md
    │
    ├── ② Journal（项目方反评价）
    │   ├── 逐条评估 SCAN 建议（agree / disagree / partial）
    │   ├── 附证据和论据
    │   ├── 列出执行行动
    │   └── 追加 cb-journal.md 对应轮次章节
    │
    ├── ③ Retired（退役留痕）
    │   ├── CB 驱动的文件退役记录
    │   ├── 退役原因 + 零引用核验
    │   └── 追加 retired.md
    │
    └── ④ 下一轮 SCAN → 回到 ①
```

### 3.2 反评价标尺（项目方使用，第三方审核参考）

| 标签 | 含义 | 要求 |
|------|------|------|
| **agree** | 完全接受，证据充分 | 已执行或列入计划 |
| **disagree** | 拒绝 | 必须附带反驳证据（用错标尺 / 事实错误 / 撞承重红线） |
| **partial** | 方向对，细节偏 | 说明哪些部分采纳、哪些调整 |

### 3.3 承重红线（不可协商）

以下项目方声明的红线，SCAN 建议不得触碰：
- 决策追踪系统编号连续性（铁律 10）
- diagnose prompt 永不动（保 Flash eval）
- 四态出口契约（success/gap/partial/answered）
- L0 走购买途径策略（不把 sim 数据当风险）

## 四、文档编号规范

### 4.1 SCAN 文档
```
docs/catch-ball/SCAN_DeepSeek_{NN}.md
```
- `{NN}` 为两位零填充序号（01, 02, 03, …）
- 每轮一个文件，不覆写
- 文件头包含归档元信息（轮次、日期、原始路径）

### 4.2 Catch-Ball Journal
```
docs/catch-ball/cb-journal.md
```
- 单文件，按轮追加不覆写
- 内部章节命名：`## CB-01`、`## CB-02` …
- 每轮四节：① SCAN 摘要 ② 我方反评价 ③ 行动 ④ 状态/新发现

### 4.3 退役台账
```
docs/catch-ball/retired.md
```
- 单文件，按退役日期追加
- 每条记录包含：文件路径、原职责、退役原因、可恢复性

### 4.4 报告结构模板（每轮 SCAN）

```
# DeepSeek 深度扫描评估报告（第 N 轮）

> **扫描模型**：... | **扫描时间**：... | **CB 轮次**：N
```

## 五、CB 权限约束

### 5.1 可执行操作
| 操作 | 范围 | 说明 |
|------|------|------|
| ✅ 新建文件 | 仅限 `docs/catch-ball/` | RULES.md, SCAN 文档, cb-journal.md, retired.md |
| ✅ 编辑文件 | 仅限 `docs/catch-ball/` | 已存在的 catch-ball 内文件 |

| 操作 | 说明 |

## 六、工作要点

1. **优先使用 Explore Agent**：大规模扫描使用 Explore Agent 并行刺探，主线程负责汇总和报告编写
2. **引用格式**：文件引用使用 `` `path:line` `` 格式（如 `core/tracker.py:142`）
3. **评分一致性**：同一轴在不同轮次间保持评分标尺一致，变化必须有明确理由
4. **趋势判断**：↑（改善 ≥0.5 分）、→（持平 ±0.4 分内）、↓（退步 ≥0.5 分）
5. **建议可验证**：每条建议尽可能提供验证方法（如"运行 `pytest tests/ -q` 确认全绿"）
6. **报告语言**：中文为主，代码/路径/技术术语保留英文
