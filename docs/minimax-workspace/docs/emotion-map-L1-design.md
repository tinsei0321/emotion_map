# 情绪地图 L1 模拟数据 — 设计方案

> 目标：生成西陵伍家核心主城（约 140 km²）+ 二马路历史街区（≈ 2 km²，详）的 L1 模拟数据，覆盖 3 个时间点（2025-01 / 2025-10 / 2026-06），服务于情绪云图、时间轴演化、L2 极性分析等功能的测试。

---

## 1. 坐标系处理方案

### 1.1 展示基准
- **CRS**：`EPSG:4547`（CGCS2000 / 3-degree Gauss-Kruger zone 36，CM 111°E）
- **原因**：宜昌主城位于 111.2°E ~ 111.4°E 之间，正好处于 36 带（108°E~111°E 的 3° 分带不够，标准是 36 带对应 CM 111°E），投影变形 < 1/1000
- **单位**：米（X = 北向，Y = 东向）

### 1.2 多源坐标系接入策略
| 数据源 | 原始 CRS | 转换链 |
|---|---|---|
| 高德 / 腾讯 | GCJ-02 | GCJ-02 → WGS84（反解密） → CGCS2000（椭球转换，差异忽略） → EPSG:4547 投影 |
| 百度 | BD09 | BD09 → GCJ-02 → WGS84 → CGCS2000 → EPSG:4547 |
| 美团 / 大众点评 | GCJ-02（推测） | 同上 |
| 小红书 / 微博 | WGS84（设备 GPS） | WGS84 → CGCS2000 → EPSG:4547 |
| 12345 热线 | 地址（需 geocode） | 地址 → WGS84（geocoder） → EPSG:4547 |

### 1.3 模拟数据约定
- **本批模拟数据统一生成在 EPSG:4547 米级坐标**
- 每个 Feature 带 `crs` 字段声明：`{"type": "name", "properties": {"name": "EPSG:4547"}}`
- 每个 Feature 的 `properties.source` 标注模拟来源：`dianping/meituan/xiaohongshu/weibo/12345`
- 真实数据接入时，在 ETL 层完成 CRS 转换，前端只处理米级坐标

### 1.4 是否需要人工偏移修复
**模拟阶段**：不需要，直接生成干净坐标。

**真实数据阶段**：建议建立"**偏移校核点**"机制 —— 在地图上预定义 10~20 个明显地物（火车站、市政府、长江大桥桥头）的真实 WGS84 坐标作为校准点。数据接入时自动检测偏移，超过 50m 触发人工校核。生产环境不推荐纯人工修复（量大易错），推荐"自动转换 + 异常点人工校核"。

## 2. 4×5 矩阵设计

### 2.1 4 个领域（domain）
| Code | 名称 | 关注重点 |
|---|---|---|
| urban_planning | 城市规划（设计） | 新建项目、控规调整、用地性质、重大设施选址 |
| urban_renewal | 城市更新 | 老旧小区改造、历史街区保护、棚改、工业遗存 |
| urban_operation | 城市运营 | 商业运营、夜经济、文旅活动、市集、节庆 |
| urban_governance | 城市治理 | 12345 投诉、噪音、拥堵、市容、应急、防汛 |

### 2.2 5 类要素（element）
| Code | 名称 | 子类举例 |
|---|---|---|
| facility | 设施 | 交通（地铁/公交/BRT）、能源、水务、环卫设施 |
| environment | 环境 | 公园、绿化、空气质量、滨江景观、山体生态 |
| service | 服务（功能） | 政务服务、医疗、教育、社区服务、便民商业 |
| culture | 文化 | 博物馆、非遗、文创、老字号、历史建筑、节庆 |
| event | 事件 | 大型活动、突发事件、舆情热点、群体行为 |

### 2.3 矩阵交叉（20 个组合）的内容倾向
模拟数据按矩阵的 20 个组合分配权重，确保每个领域 × 每个要素都有覆盖，但密度不同（如 `urban_operation × event` 在 T3 高，`urban_governance × facility` 在 T2 高）。

## 3. L1 属性字段 Schema

```json
{
  "id": "f001-2025-01-15-0001",
  "type": "Feature",
  "geometry": {
    "type": "Point",
    "coordinates": [东向坐标_米, 北向坐标_米]
  },
  "properties": {
    "source": "dianping | meituan | xiaohongshu | weibo | 12345",
    "created_at": "2025-01-15T10:23:00+08:00",
    "text": "评论原文",
    "user_archetype": "local | tourist | worker | student | migrant",
    "domain": "urban_planning | urban_renewal | urban_operation | urban_governance",
    "element": "facility | environment | service | culture | event",
    "sub_element": "transport.green | park.riverside | retail.food | ...",
    "poi_hint": "公交站 | 公园 | 商场 | 餐厅 | ...",
    "address": "逆地理编码后的地址",
    "polarity_hint": "positive | negative | neutral | mixed",
    "intensity": 1-5,
    "topic_keywords": ["关键词1", "关键词2"],
    "language": "zh-cn",
    "geocode_confidence": 0.0-1.0
  }
}
```

### 3.1 L2 可用性锚定
- `polarity_hint` 字段直接作为 L2 极性打分的种子，避免 L2 推断完全黑盒
- `intensity` 作为 L2 聚合时的权重
- `topic_keywords` 作为 L2 主题聚类的输入
- 在 3 个时间点之间，**主动设计 hint 变化**，让 L2 输出"可观察"的差异

## 4. 点位生成算法

### 4.1 POI 种子法
基于宜昌核心主城真实地理结构：

1. **核心商圈**（高密度）：解放路步行街、CBD 万达广场、铁路坝、夷陵广场
2. **老城生活**（中-高密度）：二马路、大南门、献福路、樵湖岭
3. **滨江带**（中密度）：滨江公园、长江沿线、磨基山
4. **居住区**（中-低密度）：伍家岗各大型社区、东山花园
5. **文旅节点**（中密度）：三峡博物馆、滨江公园、东山公园

每个种子点配置：
- `center_xy`：EPSG:4547 坐标
- `radius`：影响半径（米）
- `weight`：采样权重（与人流密度成正比）
- `domain_element_bias`：4×5 矩阵的偏好分布

### 4.2 边界过滤
- 长江水面：x < 5km 或 y < 0 的区域排除（已根据宜昌实际地理调整）
- 西北部山体：标定山地边界多边形排除
- 行政区外：剔除不在西陵区+伍家岗区核心主城范围的数据

### 4.3 二马路详化
二马路区域 2 km² 内独立配置：
- 70 个 POI 种子（餐厅、老字号、文创店、博物馆、老社区）
- 采样密度为核心主城平均的 **5 倍**
- 主动埋设"城市更新"演化点（T1 负面 → T3 正面）

## 5. 时间轴数据设计

### 5.1 三组数据对比
| 维度 | T1 (2025-01-15) | T2 (2025-10-20) | T3 (2026-06-25) |
|---|---|---|---|
| **季节** | 冬季 / 春节前夕 | 秋季 / 国庆后 | 夏季 / 汛期 |
| **总量** | ~600 点 | ~800 点 | ~700 点 |
| **热点区域** | 商圈 + 老社区 | 办公区 + 文旅 | 滨江 + 夜间经济 |
| **主导话题** | 年货、春运、年夜饭 | 国庆余热、银杏季 | 夜经济、防汛、高温 |
| **极性分布** | 正 60% / 负 25% / 中 15% | 正 50% / 负 30% / 中 20% | 正 55% / 负 30% / 中 15% |
| **典型事件** | 火车站春运吐槽 | 网红打卡点争议 | 滨江夜市/防汛内涝 |
| **二马路焦点** | 老旧小区改造争议 | 文创市集、节庆 | 改造完成、文旅复苏 |

### 5.2 L2 可观察变化（埋点）
- **T1 → T2**：二马路从负面 65% → 负面 40%（城市更新推进）
- **T2 → T3**：滨江带夜经济正面上扬
- **T1 → T3**：同一 POI（例：西陵二路某老旧小区）的 polarity_hint 从 `negative` → `positive`，用于演示极性迁移

### 5.3 舆情事件生命周期（T2 内部）
- 中期埋设一个"网红打卡点争议"事件：T2 早期中性 → T2 中期负面高峰 → T2 后期转为 mixed
- 让数据能演示"热点 → 冷点"的完整生命周期

## 7. 真实数据接入 API 参考

### 7.1 百度相关
- **百度地图开放平台**：https://lbsyun.baidu.com/
  - 静态图 API、热力图（自有数据上传）、地点检索 API、路线规划 API
  - 百度 POI 检索：https://lbsyun.baidu.com/faq/api?title=webapi/guide/webservice-placeapi
- **百度慧眼**（商业）：https://huiyan.baidu.com/ — 城市人口热力、出行 OD、商圈分析（需企业资质）
- **百度迁徙大数据**（公开版）：https://qianxi.baidu.com/ — 城市间迁徙热度

### 7.2 高德相关
- **高德开放平台**：https://lbs.amap.com/
  - POI 搜索、地理编码、行政区查询、路径规划
  - **JS API 自带热力图插件**（AMap.HeatMap）：前端可视化层使用，**数据需要自己提供**
- **高德慧行 / 智慧交通**（商业）：城市交通态势、出行热度

### 7.3 POI 数据
- **OpenStreetMap（OSM）**：免费，全球覆盖，宜昌有较完整数据
  - Overpass API：https://overpass-api.de/
- **POI 工厂**等数据服务商（合规需注意）

### 7.4 12345 热线数据
- 需政府合作或公开数据集，不同城市差异大
- 宜昌市政务服务热线公开数据可从宜昌市政府数据开放平台查询

### 7.5 推荐接入顺序
1. **第一阶段（模拟）**：使用本批次模拟数据
2. **第二阶段（混合）**：接入高德 POI + 自有评论数据
3. **第三阶段（完整）**：接入百度慧眼/高德慧行 + 12345 + 多平台爬虫

## 8. 数据使用提示

### 8.1 前端加载建议
- 单文件 GeoJSON 控制在 5MB 以内（T1/T2/T3 核心主城约 2~3MB）
- 二马路详情可独立加载，按视图层级（L0 全国 → L1 主城 → L2 二马路详情）按需加载
- 时间轴切换时建议预加载 3 个文件，按需切换

### 8.2 测试场景建议
- **L0 概览**：加载 T2（数据量最大），观察整体情绪云图
- **L1 时间轴**：依次加载 T1/T2/T3，演示极性变化
- **L2 二马路**：单独加载 ermalu-detail 系列，演示高密度聚合 + 主题演化
- **压力测试**：同时加载全部 6 个文件，验证前端渲染性能
