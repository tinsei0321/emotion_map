# 产品需求文档 (PRD)

> 情绪地图 (Emotion Map) — 基于多源社交数据的城市情绪空间分析平台
> 版本：v1.0 MVP | 更新日期：2026-06-15

---

## 一、产品概述

### 1.1 产品愿景

让城市规划者能"看见"市民情绪的空间分布——用数据替代直觉，用地图承载叙事。

### 1.2 产品定位

基于多源社交数据（大众点评/美团/小红书/微博/12345）的情绪空间分析平台。通过 NLP + GIS 技术，将市民的文本评价转化为情绪分数，映射到地理空间，为城市规划与治理提供数据驱动的决策支持。

### 1.3 一句话描述

**把市民的网上声音变成地图上的情绪信号，辅助城市更新与治理决策。**

### 1.4 核心价值主张

| 传统方式 | 情绪地图 |
|----------|----------|
| 问卷调查（成本高、样本少） | 社交数据自动采集（零打扰、覆盖广） |
| 经验判断（主观、不可量化） | 数据驱动（情绪分数 + 空间分布，可量化对比） |
| 静态报告（一月一报） | 动态地图（实时/准实时，随时可查） |
| 全局概要（无空间颗粒度） | 精确定位（具体到街道、POI、网格） |

## 二、目标用户与画像

### 2.1 核心用户

| 用户角色 | 典型单位 | 核心诉求 | 优先级 |
|----------|----------|----------|--------|
| **城市规划师** | 规划院/更新办 | 快速识别"情绪洼地"，辅助更新优先级排序 | P0 |
| **城市管理者** | 住建局/城管局 | 宏观掌握市民满意度空间分布，辅助政策评估 | P0 |
| **社区工作者** | 街道办/社区 | 了解本辖区居民情绪热点，精准发现问题 | P1 |
| **商业分析师** | 商业地产/品牌 | 评估商圈口碑、选址情绪评估 | P2 |
| **城市规划研究者** | 高校/智库 | 研究方法验证、数据对勘、空间分析实验 | P2 |

### 2.2 用户使用场景

1. **年度更新计划编制**：「全市 32 个更新单元，哪个舆情最差、最急需更新？」
2. **日常舆情监测**：「最近一个月，西陵区哪些地方投诉在增加？」
3. **项目效果评估**：「老旧小区改造完成半年后，居民满意度变化如何？」
4. **应急响应**：「台风/暴雨后，哪些区域的负面情绪飙升？」

### 2.3 用户技术能力假设

- 具备基本的地图浏览能力（缩放/拖拽）
- 了解城市规划基本概念（更新单元、行政区划）
- 不要求编程能力，所有操作通过 GUI 完成

### 3.1 功能全景

```
情绪地图 v1.0 MVP 功能全景

├── 情绪分析 (L2)

├── 空间可视化
│   ├── 全屏交互式地图（天地图 WMTS 底图）

│   ├── 热力图（冷热分布 + 极性加权）
│   ├── 边界叠加（多文件支持，颜色/粗细可调）

├── 成果输出

│   └── 统计图表（Altair 柱状图/饼图）

└── 辅助功能
    ├── 分析控制台（?page=console 路由）
    ├── 设计系统展示页（?page=design-system）
```

## 四、用户体验需求

### 4.1 交互流程

```
打开应用 → 全屏地图
    │
    ├── [R] 范围选择 → 上传/选择边界文件 → 边界叠加到地图
    │
    ├── [D] 数据源 → 选择 L1/L2 分析结果文件 → 数据概览 → 加载到地图
    │
    ├── [A] 分析面板 → 配置参数 → 运行分析 → 结果展示
    │
    ├── [Map] 底图切换 → 5种底图可选
    │
    ├── [LM]/[LB] 亮度模式 → Dark/Light 底图
    │
    └── [?] 图例 → 五级极性说明
```

### 4.2 界面原则

1. **地图最大化**：地图占屏幕 100%，所有控件浮动叠加
2. **渐进式展示**：先看地图整体，需要细节时再展开面板
3. **操作可回退**：每个弹窗有确定/取消，按钮可切换
4. **状态可感知**：加载有进度，空数据有引导，错误有说明

### 4.3 视觉原则

- 情绪配色语义明确：绿→蓝→灰→橙→红（正面→负面）
- Design Token 驱动：150+ Token，Light/Dark 双主题
- 光晕 + 描边确保在卫星底图上可见
- ASCII 替代 emoji（`[OK]` `[WARN]` `[LOAD]` `[ERR]`）

## 五、非功能需求

### 5.1 性能要求

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| 地图首次加载 | < 3 秒 | frontend/ MapLibre 页面完全渲染（遗留 Streamlit 同口径） |
| 1000 点渲染 | < 2 秒 | MapLibre circle 层批量渲染（遗留 Folium marker 同口径） |
| 5000 点热力图 | < 3 秒 | MapLibre heatmap 层（遗留 HeatMap 同口径） |
| L2 分析 1000 条 | < 10 秒 | SnowNLP pipeline 运行时间 |
| LLM 分类 50 条 | < 30 秒 | DeepSeek API 并发 5 条 |
| CSV 导出 10000 行 | < 1 秒 | Pandas to_csv |
| 边界文件加载 (.shp) | < 2 秒 | GeoPandas read_file |

### 5.2 兼容性

| 维度 | 要求 |
|------|------|
| 操作系统 | Windows 10/11 优先，macOS/Linux 兼容 |
| Python 版本 | 3.14.5 |
| 浏览器 | Chrome/Edge 最新版（MapLibre GL JS + 遗留 Streamlit 均依赖现代浏览器） |
| 屏幕分辨率 | 1920×1080 及以上（推荐的规划师工作站配置） |
| 编码 | UTF-8，Windows GBK 兼容通过 `_safe_print()` |

### 5.3 数据安全

- 分析结果输出禁止包含用户名/用户ID
- 天地图 Key 不硬编码（当前待迁移至 .env）
- DeepSeek API Key 不提交至代码仓库
- 原始数据本地存储，不上传云端

### 5.4 可维护性

- 代码模块化：core/ + apps/ + SCRIPT/ + SCRAPER/ + design/
- 决策追踪：所有关键函数和分支有追踪 ID
- 文档体系：PRD + 架构 + ADR + TODO + 开发笔记
- Agent 协作：11 个专业 Agent，SOP 标准化流程
