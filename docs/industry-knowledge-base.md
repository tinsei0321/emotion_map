## 一、项目设计哲学（全项目纲领，详见 [CLAUDE.md](../CLAUDE.md)「项目设计哲学」节）

**归因底层逻辑 = 「政策→情绪→项目」闭环**（情绪地图核心价值）：

```
国家宏观政策/标准/要求（顶层方向：该领域"应该"怎样）
        ↓ 指引
情绪地图归因（用市民情绪数据，定位"实际"哪里好/差、痛点在哪）
        ↓ 指向
具体「项目/事情」（落到可操作的工程/治理/规划项目）
```

情绪地图的价值 = 在**宏观政策**与**具体项目**之间，用市民情绪数据"牵线归因"——政策要求关注民生，情绪地图用数据找到民生痛点的具体空间位置，指向具体可操作项目。这是演示逻辑链"识别具体城建/更新问题"的根基。

6 条原则：① 4×5 是**归因落点矩阵**（非指标分类清单），跨领域×跨要素**多归属**；② 归因 = 政策→情绪→项目闭环；③ 官方话语对齐 + **补盲区**（事件瞬时空间影响）；④ 知识库全项目可成长、矩阵骨架稳定；⑤ 城市规划 = **城市规划设计全谱**；⑥ 多归属主/次约定。

## 三、4×5 矩阵多归属映射（知识库与归因矩阵的关系）

知识库不是"驱动要素裂分"，而是**围绕稳定的 4×5 矩阵做多对多映射**：每领域 `MATRIX_MAPPING` 记录该领域权威知识 → 多个 `(domain, element, role)` 格（role∈{主,次}）。一个现象落多格是 feature（如交通建设 = 规划×设施 = 更新×设施 = 运营×服务）。

`get_matrix_mapping(domain_key)` 查单领域映射；`all_matrix_mappings()` 反查"某矩阵格被哪些领域覆盖"（数据模拟逆推/归因展示用）。

## 四、全项目贯穿点（这套哲学不只服务 EMC）

- **演示逻辑链**（张力图面→点击→分析→识别问题）：识别的"问题" = 4×5 矩阵归因（多归属）+ 知识库官方术语。
- **目标客户画像**：业内同行/住建局/城运集团用官方话语，项目须对齐（知识库支撑）。
- **数据模拟逆推**：L2 演示目的 → 4×5 矩阵分布 → 空间落点；`MATRIX_MAPPING` 指导"某领域知识落到哪些矩阵格"。
- **Overview/前端归因展示**：饼图/4×5 矩阵/关键词高亮——矩阵多归属的可视化。
- **EMC**：`industry_kb_text(domain_key)` 渲染（v1 预留，future 按 diagnose 的 domain_lens 注入权威语境）；DOMAIN_OUTLETS 已注入四领域官方框架。

## 五、做厚路径（可成长，骨架稳定）

按需追加进对应 `ai_qa/industry_kb/<domain>.py` 的 list/dict：
- **规划**：完整社区设施清单、街道设计导则细则、各地控规/修规指标、CIM 平台。
- **更新**：城市体检 76 指标（核心+非核心）、各地更新条例、完整社区试点细则。
- **运营**：生命线 7 类设施（燃气/桥梁/隧道/供水/排水/热力/管廊）详细、活动保障细则、智慧运营案例。
- **治理**：12345 接诉即办三率细则、网格化标准、每月一题机制。
- **事件领域成体系化**（补官方盲区·差异化核心）：运营/治理 × 事件的"瞬时活动 → 空间要求/优化"专题——官方标准缺位，EMC 前瞻占位。

## 六、成长机制

政策更新 → 模块 `TOP_DESIGN`/`CORE_FRAMEWORK` 更新 → `MATRIX_MAPPING` 丰富 → EMC/数据模拟/归因展示全受益；**4×5 矩阵骨架稳定**，靠映射丰富做厚。每模块 `__main__` 自检 + `tests/test_industry_kb.py` 守护 schema，保证做厚不破格式。
