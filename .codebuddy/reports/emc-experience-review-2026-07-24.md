## 〇、总体判断

**一句话**：内核方向（Smart Agent / Dumb Tool）正确，但落地发生了**智能配置倒挂**——Agent 被要求做大量确定性工作（选工具/填参数/绑数据），每次决策都付 LLM 延迟税；Tool 缺少让"确定性"成为可能的契约（SOP 卡/成图范式/数据认知），"dumb" 退化成了 "fragile"。同时飞轮测的是"断言通过"而非"用户价值"，且闭环三处断裂——**测量失真与机制断裂叠加，导致体验差距长期不可见、不可追、不可修**。

**三基准评分**：

| 基准 | 评分 | 关键依据 |
|------|------|----------|
| 日常真实分析质量 | 2/10 | 工具例 14/15 超时无回答（E1）；上传数据识别薄弱（E3-E5）；假完成存在 |
| 演示链效果 | 4/10 | 单链路演示可通（TOL-02 通过），但稳定性不足以支撑可靠演示 |
| 架构内核一致性 | 4/10 | 编排器确定性骨架在（plan-once-execute/谓词/出口），但智能摊薄+数据认知缺位，内核只实现了一半 |

**证据索引**（全文引用）：
E1 工具例 14/15 超时（report-01）；E2 超时例总耗时 91~137s（report-01 耗时列）；E3 addLayer 零去重（state.js:667-707）；E4 字段样本 6字段×2值×24字符且仅可见层（tools.js:466-482,519-523）；E5 层语义靠层名正则（tools.js:486-495）；E6 _userPinned 新话轮不复位（panel.js:1535/1547）；E7 字段字典闲置（core/field_dictionary.py 未接前端上传链路）；E8 切片键已存在未接入寻址（main.js:152 tagAllLayers）；E9 词表漂移（prompts.py:190 vs paradigm.py:454/470）；E10 template 信号断链（schemas.py:11-22）；E11 分析型工具无 layerId 成功判定单列（harness.js:375）。

## 一、机制层面：测试流程 / 指标定义 / 迭代逻辑的局限

### 1.1 测试流程局限

| 局限 | 事实 | 后果 |
|------|------|------|
| 90s 上限混淆因果 | 14/15 失败全是"s3 回答超时"（E1），不是答错而是**没答** | 报告首要暴露的是延迟，正确性信号被淹没——测试在替产品架构问题背锅 |
| 串行 runner 无预算概念 | llm 例串行、例间固定等待；200 例全量需 1.5-3h | 全量回归成本高 → 实际只跑小样本 → 样本偏斜 |
| slider 头部切片偏斜 | 截取 = `cases.slice(0, limit)`，类别排序 CPD/UI/PRED 在前 | 小 limit 时全是 no-llm，llm 类几乎抽不到——"抽 25 例"给的是覆盖面假象 |
| 信号采集断链 | template 从请求体抓取，但 ChatRequest 无 diagnose 字段（E10） | 意图断言 tmplOk 恒 false，意图测试实际只剩工具信号半残运行 |

### 1.2 指标定义局限

| 指标问题 | 事实 | 后果 |
|----------|------|------|
| pass 指标失真 | 参数正确性 10 例断言恒 `pass:true`（expect* 定义未接断言）；意图断言 `tmplOk \|\| toolOk` 软断言 | 覆盖率数字虚高，"覆盖假象" |
| 无延迟/成本指标 | 报告无 p50/p95、无 LLM 调用数、无 token 统计 | 头号体验杀手（慢）在指标体系里不存在——不测就不会被优化 |
| 无 UX 指标 | 无首字节时间、无首图层时间、无落图率、无假完成率 | 用户核心价值（以图说话）无度量载体 |
| 判定混淆 | "测试判断正确"把系统错误与断言误杀/漏判混为一格 | 迭代时不知道修 prompt 还是修断言 |
| 出口指标缺失 | 四态出口（result/gap/partial/drift）分布不统计 | 假完成（partial 报成 result）无暴露通道 |

### 1.3 飞轮迭代逻辑局限

```
现状闭环：跑 → 报告（md 表格） → 人工看 → ……断
应有闭环：跑 → 报告 → 失败聚类 → 回流 prompt 池/SOP 负例/few-shot → 修改 → 回归 gate → 基线前移
```

| 断裂点 | 事实 | 后果 |
|--------|------|------|
| 投票不落盘 | 行内 V/X 只存内存 `_results`，刷新即丢 | 人工判定资产无法积累 |
| 失败不回流 | 无 (prompt, 期望, 实际) 三元组结构化产出 | prompt 池调优靠记忆不靠数据 |
| 无回归 gate | 修改后无"pass 率不得降"的自动门槛 | 改了 A 破 B 不可见 |
| 无 golden set | episodes.jsonl 63+ 条真实会话未回放标注 | 用例分布 = 想象分布，非真实分布 |
| 遥测孤岛 | harness 已有 template 命中率 localStorage 遥测 + 80% gate | 数据滞留浏览器，与飞轮/报告零连通——半成品闭环 |

**小结**：飞轮的"轮"（跑→报告）成型，"飞"（迭代放大）未成型；且轮子测量的对象（断言通过）与目标（用户体验）之间存在两层失真（信号失真 + 分布失真）。

## 二、体验层面：操作痛点 / 功能缺失 / 预期落差

### 2.1 四痛点的证据化确认

| 用户痛点 | 证据 | 直接原因 |
|----------|------|----------|
| 效率很低，回答很慢 | E1/E2：工具例 91~137s | 串行管道最多 5 次 LLM 调用（diagnose Pro + agent N轮 + answer Pro + review + revise Pro），两个 reasoner 串行即吃光 90s |
| 不能识别上传数据 | E3 重复堆叠 / E4 薄样本 / E5 层名正则 / 隐藏层对 EMC 不可见（tools.js:522） | 系统对数据的认知 = "带名字的图层"，无字段级/语义级清单 |
| 基本 GIS 操作经常出错 | E9 词表漂移（prompt 教 LLM 选不存在的 template）；report-03 未触发 density | 一次性直选工具，无方法→尺度→工具链式结构，无参数确定性校验 |
| 复杂任务只做一半/假完成 | ReAct 上限 8 轮打满即停；EXIT_RESULT 裁定 successObs>0 即算成；narrated 修复 ≤1 轮后 GAP | 出口裁定与"用户要的结果"不对齐 |

**第五痛点（本轮新证实）**：对话窗口不跟随——上滑一次后 `_userPinned` 永久置位，新话轮不复位（E6），用户必须手动点回底按钮。

### 2.2 功能缺失清单

| 缺失 | 影响 |
|------|------|
| 无阶段进度反馈、不可取消 | >90s 黑箱等待 = "卡住"感，体验崩塌的主观感来源 |
| 无上传去重、无字段确认向导 | 图层堆叠污染上下文；字段误识别无纠正通道 |
| 回答不引用图层（无图层芯片/落图高亮） | "以图说话"无 UI 契约，图与文两张皮 |
| 无成本/耗时可见性 | 与"调用次数优先"承重脱节，用户与开发者都无感知 |
| 无数据预检呈现 | "缺数据"靠 LLM 猜（data_plan），不靠预检报告说话 |

### 2.3 预期落差的结构

| 维度 | 用户预期 | 系统现状 |
|------|----------|----------|
| 角色 | 日常分析工具（数据驱动/尺度诚实/回答有用） | 按演示链调优的对话壳 |
| 数据 | 上传即可问 | 上传后 LLM 看不见（隐藏层）/看不懂（薄样本）/重复看（无去重） |
| 交互 | 一句话出图，过程有动静 | 90s 黑箱后超时或无图 |
| 结果 | 完成状态诚实（做完/没做完说清楚） | 部分完成也可能报"完成" |
| 复杂任务 | 多步链全程走完 | 8 轮打满半途停 |

## 三、根因剖析：机制设计与真实场景的五个脱节点

### 脱节 1：智能配置倒挂（内核只实现了一半）

Smart Agent 的智能应集中在**少数高价值决策**（意图理解、方法选择、结果研判），其余全部确定性。现状相反：

- Agent 侧：工具选择、参数填充、数据绑定这些**确定性可解**的工作全交给 LLM——每次决策付一次延迟税+一次犯错概率；
- Tool 侧：没有 SOP 卡（用途/适用尺度/前置条件/参数契约/失败模式）、没有成图范式、没有产出自检——"dumb" 所需的富契约缺失，于是 dumb 变成 fragile；
- 确定性基础设施**已存在却未接线**：字段字典（E7）、切片键（E8）、SKILL_DEFS 参数契约、plan-once-execute 骨架。**LLM 在猜代码本可知道的事**——这是最昂贵的一种架构浪费。

### 脱节 2：数据认知缺位

系统把数据当"带名字的图层"（层名正则判语义，E5），用户把数据当"有意义的数据集"（有字段语义、时点、质量、覆盖范围）。中间缺一个**导入时一次性计算、全链路复用的数据清单（manifest）**。没有它：diagnose 的 data_plan 靠猜、grounding 靠薄样本（E4）、参数校验靠运气、去重无从谈起（E3）。

### 脱节 3：延迟架构错位

同步串行管道（diagnose→agent→answer→review→revise）是"深度研究"型架构，用户要的是"交互工具"型体验（秒级有动静、渐进出结果）。模型策略全程 Pro reasoner 把每次交互当深度研究做；无预算、无降级、无进度、无取消。**90s 上限不是测试参数，是产品架构的镜像**。

### 脱节 4：成功定义错位

系统成功 = `successObs>0 或 newLayerCount>0`；用户成功 = "我要的那张图/那个数出现了吗，完成状态说实话了吗"。以图说话无架构支撑：产物不是一等公民（回答不引用图层）、落图无自检（空结果也可能被叙述成完成）、partial 与 result 的裁定线过宽。**假完成不是 LLM 撒谎，是裁定器定义错了成功**。

### 脱节 5：测试与价值错位

- 指标失真：断言半残（E10）+ 恒 pass（参数）+ 软断言（意图）→ pass 率不代表正确性；
- 分布失真：用例来自想象的演示链分布，用户真实分布（自己的数据/自己的问题/复杂链/负例）覆盖为零；
- 迭代失真：闭环断在报告（§1.3），pass 率数字本身都不能持续追踪。
- 结论：**飞轮目前测不出体验差，更修不了体验差**——它需要先修自己（plan 文档 M1-M3），才能成为迭代引擎。

### 根因收敛图（痛点 → 四层归因）

| 痛点 | 表象 | 直接机制根因 | 架构根因 |
|------|------|--------------|----------|
| 慢 | 91~137s 无回答 | 串行多 Pro 管道 | 脱节 3：延迟架构错位 |
| 不识别上传 | 堆叠/看不见/看不懂 | 无去重+薄样本+可见性纪律 | 脱节 2：数据认知缺位 |
| GIS 常错 | 选错工具/填错参/不触发 | 一次性直选+词表漂移 | 脱节 1：智能倒挂（契约缺失） |
| 半途停/假完成 | 8 轮打满/partial 报 result | 出口裁定宽松+无落图自检 | 脱节 4：成功定义错位 |
| 测了没改善 | pass 率与体验无关 | 信号断链+闭环断裂 | 脱节 5：测试与价值错位 |

## 四、问题清单（分级，全部挂证据）

**H 级（阻断体验/测量）**
- H1 延迟架构：串行多 Pro 管道无预算无降级无进度（E1/E2）
- H2 数据认知缺位：无 manifest，LLM 猜代码可知之事（E3/E4/E5/E7/E8）
- H3 智能倒挂：工具选择/参数绑定未确定性化，Tool 无 SOP 契约（E9）
- H4 成功定义错位：出口裁定宽+无落图自检+回答不引用图层（E11）
- H5 飞轮信号失真：template 断链+参数恒 pass+软断言（E10）
- H6 飞轮闭环断裂：投票/回流/gate/golden set/遥测五断点

**M 级（显著磨损）**
- M1 滚动不跟随（E6）｜M2 无进度无取消｜M3 无去重 UX（E3）｜M4 无成本可见｜M5 用例分布偏演示链｜M6 slider 抽样偏斜

**L 级（抛光）**
- L1 报告无 JSON/元数据｜L2 谓词测试两处重复｜L3 confirm 阻塞存报告
